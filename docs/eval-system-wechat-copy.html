我是怎么给 Agent 搭建 Eval 评测体系的

Agent 的质量很难靠感觉维护。我的做法是把关键能力拆成可运行的 Eval：能确定的用程序断言，语义质量交给 grader LLM，所有结果落到 JSONL，再由 CLI 做查看、汇总和通过率门禁。

问题不是“模型聪不聪明”，而是“行为能不能回归”

做 Agent 之后，很快会遇到一个工程问题：改了 prompt、工具、上下文压缩、provider 适配之后，怎么判断系统有没有退化？

普通单元测试能覆盖确定性代码，但覆盖不了模型行为。人工试几轮又太主观，而且很难留下可比较的记录。我的目标不是做一个排行榜，而是给每次 Agent 迭代建立行为验收边界。

所以这套 Eval 体系有三个基本要求：

  • 必须能像普通测试一样被组织、选择和编译。
  • 必须能检查真实行为，比如 tool call、文件写入、go test 结果。
  • 必须能记录每条 rubric 的输出、理由、分数和 pass/fail，便于追踪。

整体架构：Go test 是执行器，JSONL 是事实表

Eval 用例仍然是 Go 测试，放在被评估包旁边：

pkg/evals/                         # 通用评测框架
pkg/agent/agent_eval_test.go       # Agent 基础行为
pkg/agent/tool_eval_test.go        # Agent 工具调用行为
pkg/coding_agent/coding_eval_test.go
cmd/modu_eval/                     # 运行、查看、CI 汇总

运行链路是这样的：

  1. 测试调用 evals.Run(t, name, func(e *EvalT){...})，内部拿到模型 provider、model、grader。
  2. 用例驱动真实 agent.Agent 或 coding_agent.CodingSession，产生消息、工具调用和文件副作用。
  3. 确定性行为用 Go 断言；语义质量调用 grader LLM，要求返回结构化 JSON。
  4. 每条判断都通过 RecordScore 追加到模块根目录的 evals.jsonl。
  5. cmd/modu_eval 负责运行测试、打开 TUI、输出 CI summary、按 pass rate 做门禁。

evals.Run：把 Go test 变成多模型评测器

核心入口在 pkg/evals/evals.go。它没有重新发明测试框架，而是在 Go test 上面加了一层模型注入和运行次数控制。

type EvalT struct {
    *testing.T
    *Eval
}

type Eval struct {
    Provider     providers.Provider
    Model        *types.Model
    Grader       providers.Provider
    GraderModel  *types.Model
    ProviderSpec ProviderSpec
    GraderSpec   ProviderSpec

    runNumber int
}

EvalT 同时持有 testing.T 和当前评测上下文，所以 Eval 用例里既能 e.Fatalf，也能访问 e.Provider、e.Model、e.Grader。

func Run(t *testing.T, name string, f func(e *EvalT)) {
    numEvals := NumEvalsOrSkip(t)
    specs := providerSpecsFromEnv()

    for _, spec := range specs {
        eval, _ := NewEval(spec)
        testName := fmt.Sprintf("[%s/%s] %s", spec.ProviderID, spec.ModelID, name)
        t.Run(testName, func(t *testing.T) {
            e := &EvalT{T: t, Eval: eval}
            for i := 0; i < numEvals; i++ {
                e.runNumber = i
                f(e)
            }
        })
    }
}

这里有几个关键点：GOEVALS 控制是否运行和重复次数；EVAL_PROVIDER 支持逗号分隔；子测试名包含 provider/model；runNumber 会记录到结果里，用来分析稳定性。

ProviderSpec：统一走 OpenAI-compatible 接口

Provider 注入在 pkg/evals/provider.go。我没有为每个模型厂商写一套 Eval 逻辑，而是把评测 provider 收敛成一个 ProviderSpec：

type ProviderSpec struct {
    ProviderID string
    BaseURL    string
    APIKey     string
    ModelID    string
}

NewProvider 统一创建 OpenAI-compatible provider：

return openai.New(
    spec.ProviderID,
    openai.WithBaseURL(spec.BaseURL),
    openai.WithAPIKey(spec.APIKey),
), nil

grader 也独立配置。如果没有设置 GRADER_*，grader 会复用被测模型。这样本地评测可以全用 LM Studio，严肃评测可以用更强模型做 grader。

为什么不全靠 LLM 评分

这套体系里有一个很重要的分层：确定性判断和语义判断分开。

evals.ContainsT(e, "26", output)
evals.NotContainsT(e, "a - b", src)
evals.ToolCalledT(e, messages, "get_weather")
evals.AssertT(e, "go test ./... passes", out, passed)

这些断言不会调用 grader LLM，但仍然会写入 evals.jsonl。真正需要语义理解的部分才走 LLMRubricT：

evals.LLMRubricT(e,
    "回答转述了工具返回的天气（26度、晴），没有编造其他天气数据",
    output,
)

LLMRubric：让 grader 只返回 JSON

pkg/evals/graders.go 里，grader prompt 被限制成只返回一个 JSON 对象：

{"reasoning":"short explanation","score":0.0,"pass":false}

判分过程有几个工程细节：

  • 每次 grader 调用有 60s timeout，避免卡死整轮测试。
  • provider 或 JSON parse 失败最多重试 3 次。
  • temperature 固定为 0.0，减少评分漂移。
  • MaxTokens 给到 4096，避免 reasoning 模型还没吐出 JSON 就被截断。
  • 会剥离某些 reasoning 模型输出里的 <think>...</think>。
  • 通过 extractJSONObject 从响应中截取第一个 JSON 对象，降低格式污染带来的失败率。

最终通过条件不是只看 grader 的 pass=true，还要分数达到统一阈值：

verdict := result.Pass && result.Score >= passScoreThreshold

RecordScore：JSONL 是评测事实来源

每条断言或 rubric 都会写一行 JSONL。结构在 pkg/evals/record.go：

type EvalLogLine struct {
    Name      string  `json:"name"`
    Timestamp string  `json:"timestamp"`
    RunNumber int     `json:"run_number"`
    Provider  string  `json:"provider"`
    Model     string  `json:"model"`
    Grader    string  `json:"grader"`
    Rubric    string  `json:"rubric"`
    Output    string  `json:"output"`
    Reasoning string  `json:"reasoning"`
    Score     float64 `json:"score"`
    Pass      bool    `json:"pass"`
}

写文件时先把整行 marshal 成一个 buffer，再用 O_APPEND 打开文件并一次 Write。原因是 go test 跑多个 package 时是多个进程。小 JSONL 行一次 append write，在常见文件系统上可以避免行被交叉写坏。

modu_eval：运行、查看和 CI 门禁

cmd/modu_eval 是这套 Eval 的外壳。它没有替代 Go test，而是包了一层命令体验。

cmdArgs := append([]string{"test"}, args...)
cmd := exec.Command("go", cmdArgs...)
cmd.Env = append(os.Environ(), "GOEVALS=1")

check 支持 --min-pass-rate，实现上是按测试名聚合 JSONL 里的 pass/total。概率性 Eval 可以多跑几次，用通过率做门禁，而不是被单次随机性绑架。

Agent Eval：不是看回答，而是看消息和 tool call

pkg/agent 的 Eval 直接驱动 agent.NewAgent。工具类 Eval 会注册一个 deterministic tool。这个工具返回固定结果，并记录调用参数：

type recordingTool struct {
    name   string
    result string
    calls  []map[string]any
}

func (t *recordingTool) Execute(..., args map[string]any, ...) (types.ToolResult, error) {
    t.calls = append(t.calls, args)
    return types.ToolResult{
        Content: []types.ContentBlock{&types.TextContent{Text: t.result}},
    }, nil
}

所以天气 Eval 不只是看回答里有没有天气，还会检查模型是否真的调用了 get_weather、工具参数里是否有 city、最终回答是否包含工具返回值，以及它有没有编造其他天气数据。

Coding Agent Eval：在临时项目里检查真实副作用

pkg/coding_agent 的 Eval 会创建临时目录，启动真实 CodingSession：

sess, err := coding_agent.NewCodingSession(coding_agent.CodingSessionOptions{
    Cwd:       dir,
    AgentDir:  filepath.Join(dir, ".coding_agent"),
    Model:     e.Model,
    GetAPIKey: func(string) (string, error) { return "", nil },
})

几个辅助函数构成了 Coding Eval 的基本验收能力：

func assertGoFileParses(e *evals.EvalT, path string) string {
    data, _ := os.ReadFile(path)
    _, err := parser.ParseFile(token.NewFileSet(), path, data, parser.AllErrors)
    if err != nil {
        e.Fatalf("%s is not valid Go", filepath.Base(path))
    }
    return string(data)
}

func runGoTest(dir string) (string, bool) {
    cmd := exec.CommandContext(ctx, "go", "test", "./...")
    cmd.Dir = dir
    out, err := cmd.CombinedOutput()
    return string(out), err == nil
}

这意味着 Coding Eval 的通过标准不是“模型说它已经完成”，而是文件真的出现在磁盘上、Go 代码真的能 parse、测试真的能通过。

隐藏测试：防止模型只满足提示里的样例

Coding Agent 很容易写出“看起来对”的代码。为了避免模型只过拟合显式样例，我在一些 Eval 里采用隐藏测试。

  1. 临时项目里只放任务描述或公开测试。
  2. 让 Agent 实现功能。
  3. 评测代码在 Agent 完成后写入隐藏测试文件。
  4. 由评测自己运行 go test ./...。
const hiddenSlugTestSource = `package slug

func TestSlugifyHiddenCases(t *testing.T) {
    cases := map[string]string{
        "Hello, World!": "hello-world",
        "Multiple   Spaces": "multiple-spaces",
        "Already--Slug": "already-slug",
        "Trim Me ": "trim-me",
    }
    ...
}`

把项目工程约束也变成 Eval

1. 保护无关文件

我会在临时项目里放一个 notes.txt 哨兵文件，任务只要求修 add.go。完成后 Eval 断言：

data, _ := os.ReadFile(filepath.Join(dir, "notes.txt"))
evals.AssertT(e, "unrelated notes.txt is unchanged", string(data), string(data) == sentinel)

2. 新功能必须补测试

Slugify Eval 不只要求实现 slug.go，还要求创建 slug_test.go。评测会 parse 测试文件并检查包含 TestSlugify。

3. pkg 公共包变更同步 README

另一个 Eval 在临时项目里创建 pkg/mathutil，要求实现 Clamp 并更新 pkg/mathutil/README.md。验收同时检查 README 和隐藏测试。

这套体系目前覆盖了什么

pkg/agent 侧主要覆盖中文事实回答、多轮上下文记忆、结构化 JSON 输出、按需调用工具、避免无谓工具调用、多步工具链和工具错误处理。

pkg/coding_agent 侧主要覆盖创建 Go 文件和函数、根据测试实现代码、修复 bug、重构并保持测试通过、读取代码后回答事实问题、跨文件协调修改、TDD red-to-green、保护无关文件、新增实现时同步补测试，以及 pkg 公共包变更同步 README。

我的几个实现原则

  • Go test 做宿主。 不另起一套测试协议，直接复用 testing.T、t.Run、临时目录和包选择能力。
  • 确定性优先。 substring、regex、tool call、文件存在、go test 结果都由程序判断。
  • Rubric 要窄。 每条 rubric 只判断一个语义点，失败时才知道具体退化在哪里。
  • 记录每个判断。 无论是确定性断言还是 LLM 判分，都写入统一 JSONL。
  • 真实副作用优先。 Coding Agent 的 Eval 看磁盘、看 Go parser、看测试结果，不看模型自述。
  • 概率性用 pass rate。 多跑几次，用 --min-pass-rate 做门禁。

总结

Agent Eval 的价值，不是证明某个模型很强，而是让系统迭代有回归边界。

这套实现最重要的地方在于：它没有把评测做成纯 prompt 工程，而是把模型行为接回了工程世界。工具有没有真的调、文件有没有真的改、测试有没有真的过、无关文件有没有被误伤，这些都能被程序检查。

当 Eval 能做到这一点，Agent 才不只是一个“看起来能用”的聊天系统，而是一个可以持续改、持续测、持续收敛的工程系统。
